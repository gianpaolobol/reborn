# Re-born UX Bible v0.1

## UX doctrine

Re-born must feel like a repair guide, not like a marketplace.

The user should never feel forced to know technical language before the platform has helped them clarify the repair problem.

---

## Primary UX promise

At every step the user must understand:

1. what Re-born has understood;
2. what Re-born is not sure about;
3. what information is missing;
4. what the next best action is;
5. what options are available.

---

## Core flows

### Flow A — Non-technical repair user

1. landing;
2. start repair;
3. upload photos;
4. describe issue;
5. AI/manual interpretation;
6. guided questions;
7. repair path selection;
8. provider/model/maker action;
9. outcome confirmation.

### Flow B — Maker contribution

1. maker onboarding;
2. upload repair model;
3. component mapping;
4. verification;
5. publication;
6. repair usage;
7. royalty/credit feedback.

### Flow C — Provider fulfilment

1. provider onboarding;
2. capability setup;
3. receive request;
4. quote;
5. fulfil;
6. confirm;
7. receive rating/payment.

### Flow D — Admin assisted intelligence

1. review cases;
2. correct classification;
3. approve models;
4. resolve disputes;
5. improve taxonomy;
6. inspect learning signals.

---

## Tone of voice

Use language that is:

- precise;
- calm;
- reassuring;
- transparent;
- technical only when needed.

Avoid:

- hype;
- fake certainty;
- crypto/marketplace slang;
- too much 3D printing jargon;
- “magic AI” language.

---

## Confidence UX

AI results should be presented as:

- “We think this could be…”;
- “Confidence: high / medium / low”;
- “To be more accurate, we need…”;
- “Please confirm…”;
- “This result is not verified yet.”

Never present uncertain recognition as fact.

---

## Marketplace UX rule

Marketplace results should appear only after the repair context is clear.

Bad:

> Search all STL files.

Good:

> We found 3 possible replacement models for this component.
