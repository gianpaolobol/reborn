# MVP Scope

## MVP thesis

The first version must prove the repair loop, not the entire platform.

The MVP should be narrow, guided and outcome-oriented.

---

## Included

### Repair case intake

- title;
- description;
- photos;
- known object data;
- missing component flag;
- damage type;
- measurements;
- location if provider routing is needed.

### Repair classification

- category;
- product;
- brand;
- model;
- component;
- confidence;
- notes;
- missing data.

### Repair paths

- existing model;
- provider print request;
- maker request;
- AI generation placeholder;
- spare part link placeholder;
- manual review.

### Maker/provider basics

- profiles;
- capabilities;
- model metadata;
- requests;
- quotes.

### Outcome feedback

- success/failure;
- rating;
- photos;
- notes;
- fit feedback.

---

## Excluded

- full AI model training;
- full 3D viewer/editor;
- full e-commerce;
- automatic shipping integration;
- complex enterprise permissions;
- native app;
- advanced wallet ledger.

---

## MVP quality bar

The MVP must feel like a repair assistant, not a generic upload form.

Every feature must be connected to the Repair Journey.
